⚡ 自动触发机制

全链路自动化通过事件驱动和轮询机制实现各子流程之间的自动触发，确保任务从识别到验收无缝流转。

🔗 事件驱动

  • 消息接收 → 触发自动识别
  • 任务创建 → 触发自动分析
  • 分析完成 → 触发自动拆解
  • 测试失败 → 触发问题修复

🔄 轮询检测

  • 自动执行器：每 30s 轮询待执行任务
  • 自动监控：每 5min 检查任务状态
  • 自动验收：每 30min 检查待验收任务
  • 自我进化：每日 22:00 执行回顾

🎯 设计理念

通过自然语言处理技术，自动识别用户消息中的任务意图，无需用户显式创建任务。支持多种触发词和上下文理解。

  • 关键词触发 - 包含动作词、时间词、祈使词自动触发
  • 上下文理解 - 结合对话历史理解用户意图
  • 智能分类 - 自动识别任务类型和优先级
  • 双路径检测 - Gateway Hook + Agent 双重保障，不遗漏
  • 幂等去重 - 基于消息 hash 去重，避免重复创建

⚡ 双路径检测机制

为确保任务意图不被遗漏，采用双路径检测策略：

🔀 Gateway Hook（异步）

  • 消息到达时自动触发
  • 异步执行，不阻塞消息传递
  • fire-and-forget 模式
  • 失败不影响主流程

🤖 Agent 同步检测

  • Agent 收到消息后调用 API
  • 同步执行，获取检测结果
  • 可知道任务是否已存在
  • 作为 Gateway Hook 的兜底

🔒 幂等去重机制

基于消息原文 hash 去重，5 分钟窗口内相同消息不重复创建：

message_hash = MD5(message.trim().toLowerCase()).substring(0, 16)

// API 返回
{
  "success": true,
  "isTask": true,
  "taskId": "xxx",
  "alreadyExists": true  // 表示任务已存在
}

✅ 解决问题：Hook 和 Agent 可能同时检测同一消息，但不会重复创建任务

🎯 v2.0 核心改进

新增联动机制：修复队列与问题状态自动同步

操作 修复队列状态 问题状态
开始修复 pending → running open → in_progress
修复完成 running → completed in_progress → resolved
修复失败 running → failed in_progress → open（允许重试）
关闭问题 移出队列 → closed

🎯 设计理念

测试失败或发现异常时，自动进入问题修复流程，定位问题并尝试修复。

  • 问题定位 - 分析错误日志，定位问题根源
  • 自动修复 - 对于已知问题自动修复
  • 人工介入 - 复杂问题通知人工处理
  • 状态联动 - 修复队列与问题状态自动同步

🔗 衔接机制
  1. 状态恢复：通过 agent_id 查找该 Agent 的进行中里程碑（status = in_learning / in_practice）
  2. 进度恢复：读取里程碑的 learning.progress 和 practice.progress 字段
  3. 继续学习：从上次中断的知识点继续，跳过已完成的部分
  4. 时间计算：累计学习时长，更新 learning.hours
  5. 成果保留：已完成的 deliverables 不会丢失，可继续完成未完成的
⚠️ 中断恢复规则
中断场景 恢复策略 数据保留
Agent 下线后重启 查找该 Agent 的进行中里程碑，继续学习 进度、时长、deliverables 全部保留
学习被打断（新任务） 保存当前进度，任务完成后自动恢复 里程碑状态保持 in_learning
里程碑中途切换 支持，但建议完成当前里程碑再切换 原里程碑进度保留
学习验证失败 重新学习未通过的部分，不重置进度 保留已通过的部分

🔗 三层任务通知机制

📡 任务流转机制 v4.0（当前实际运行）

重要更新 (2026-04-05)：文档已修正为当前真实运行的流程。详见：TASK-FLOW-EVOLUTION.md

  • 第一层（正在使用 ✅）：Federation spawn - unified-task-dispatcher 调用 /api/federation/spawn
  • 第二层（未实现）：sessions_send - OpenClaw 原生机制（预留）
  • 第三层（后备 ⚠️）：轮询 - unified-task-dispatcher 定时检查 pending_assignments
⚠️ 重要：注册配置错误将导致无法接收任务！

所有使用联邦通信的 Agent 必须按照本节要求正确注册，否则任务分配后会提示"目标 Agent 不在线"。

📡 推荐：sessions_send 原生通知

1. 通知机制（v4.0 推荐）

auto-task-assigner 使用 OpenClaw 原生 sessions_send 机制通知 agent-listener：

// auto-task-assigner.js 中发送通知 await gatewayRequest('POST', '/tools/invoke', { tool: 'sessions_send', args: { sessionKey: `agent:${targetAgent}:main`, message: JSON.stringify({ type: 'task_assigned', taskId: task.id, taskTitle: task.title }) } });
  • sessionKey 格式：agent:{agentId}:main（如 agent:coder:main）
  • 消息类型：task_assigned
  • Agent listener 收到后启动 Subagent
2. Agent Listener 接收通知

agent-listener 监听 Gateway 会话消息，收到 task_assigned 后使用 sessions_spawn 启动 Subagent：

// agent-task-listener-wrapper.js 中接收通知 ws.on('message', (data) => { const msg = JSON.parse(data); if (msg.type === 'task_assigned') { // 启动 Subagent const { sessions_spawn } = await import('openclaw'); await sessions_spawn({ agentId: agentId, task: msg.taskTitle, mode: 'run', timeoutSeconds: 600 }); } });
✅ v4.0 优势
  • 使用 OpenClaw 原生机制，无需额外服务
  • 更稳定可靠，避免 agent-im-server 依赖
  • 每个 listener 有独立会话配额，避免达到限制

❓ 常见问题

问题1：Agent 显示在线但收不到任务
症状： 
agent-im-server 日志：目标 Agent 不在线: morning-coder-1:coder-1
但 curl /api/agents 显示：default:coder-1 在线
原因： INSTANCE_ID 配置错误
解决： 
# 检查当前 INSTANCE_ID
pm2 env  | grep INSTANCE_ID

# 修复
pm2 delete agent-listener-coder-1
INSTANCE_ID=morning-coder-1 pm2 start ... -- coder-1
问题2：WebSocket 连接后立即断开
症状： 
[coder-1] WebSocket 已连接
[coder-1] WebSocket 已断开
[coder-1] 5秒后重连
原因： 连接了错误的端口（Gateway 18789）
解决： 检查 wsUrl 配置，确保连接 18790 
// 正确 ✅
wsUrl: 'ws://localhost:18790'  // agent-im-server

// 错误 ❌
wsUrl: 'ws://localhost:18789'  // Gateway
